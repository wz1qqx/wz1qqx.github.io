vLLM Humming kernel 集成学习笔记

结论先行:Humming 在当前 vLLM 工作树里是一个外部 humming Python 包提供的混合精度 GEMM / MoE expert 计算后端。vLLM 本仓库负责注册、选择、权重格式转换、workspace / routing / All2All 编排,真正的算子入口是外部包的 humming.layer.HummingMethod.forward_layer()。

源码基准:/Users/f/Documents/vllm-workspace/tasks/opt-kimi-pd/vllm-int 当前工作树。本文只依据本地源码行号,不推断外部 humming 包内部 kernel 名称。

一句话定位

问题源码事实
humming kernel 是什么vLLM 中名为 humming 的 quantization / MoE backend。它把 Linear 或 MoE 的权重转成 Humming 标准格式,再调用外部 HummingMethod.forward_layer() 做 dense GEMM 或 routed MoE 的 w13/w2 expert GEMM。
对应实现文件主要适配层:vllm/model_executor/layers/quantization/humming.py; MoE expert 实现:vllm/model_executor/layers/fused_moe/experts/fused_humming_moe.py; GPT-OSS/MXFP4 backend 选择:vllm/model_executor/layers/fused_moe/oracle/mxfp4.py; 权重转换工具:vllm/model_executor/layers/quantization/utils/humming_utils.py。
在推理流程的位置MoE FFN / expert GEMM, 位于 transformer layer 的 MoE MLP 分支。它不属于 attention、KV cache、sampling; prefill 和 decode 都会在执行 MoE 层 forward 时经过同一套 expert compute 路径。
是否默认启用不是 MXFP4 auto priority 的默认候选之一。--moe-backend humming 显式映射到 Mxfp4MoeBackend.HUMMING; --quantization humming 显式让 quantization override 选择 HummingConfig。

源码入口和真实名称

代码里存在 humming 字面名,因此不需要另找别名。vLLM 对它的真实命名是:

  • quantization method: HummingConfig.get_name() 返回 "humming", 见 vllm/model_executor/layers/quantization/humming.py:185-194。
  • MoE backend: MoEBackend Literal 包含 "humming", 说明为 Use Humming Mixed Precision kernels, 见 vllm/config/kernel.py:122-137 与 vllm/config/kernel.py:171-186。
  • MXFP4 oracle backend enum: Mxfp4MoeBackend.HUMMING = "HUMMING", 见 vllm/model_executor/layers/fused_moe/oracle/mxfp4.py:60-89。
边界:本仓库没有 csrc/humming*.cu 这类自带 CUDA 实现。humming.py 在 import 阶段从外部包导入 humming.dtypes、humming.layer.HummingMethod、humming.schema、humming.utils.weight.quantize_weight, 见 vllm/model_executor/layers/quantization/humming.py:51-67。Dockerfile 中安装的是 git+https://github.com/novitalabs/humming-int.git@dev#egg=humming, 见 docker/Dockerfile:725-731。

推理流程位置

Humming 的主战场是 MoE FFN。一个典型 MoE 层先通过 router 选专家,再把 token 分派给 expert GEMM,最后按 top-k 权重合并输出。vLLM 的 modular MoE 注释把流程概括成:

[Router] -> [Quantize-Dispatch] -> [Permute-Experts-Unpermute] -> [Combine]

这段分层定义位于 vllm/model_executor/layers/fused_moe/modular_kernel.py:46-80。Humming 对应其中的 FusedMoEExpertsModular expert compute 段,具体做两次 expert GEMM:

两条集成路径

1. quantization=humming

这是 Humming 作为 quantization method 的路径。注册点在 vllm/model_executor/layers/quantization/__init__.py:

  • QuantizationMethods 包含 "humming", 见 quantization/__init__.py:12-47。
  • get_quantization_config() lazy import HummingConfig, 并把 "humming" 映射到 HummingConfig, 见 quantization/__init__.py:109-170。
  • ModelConfig._verify_quantization() 在用户显式指定 self.quantization == "humming" 时把 humming 放到 override 列表最前, 见 vllm/config/model.py:1002-1038。

HummingConfig.get_quant_method() 根据 layer 类型分发:

LinearBase     -> HummingLinearMethod
RoutedExperts  -> HummingMoEMethod
无 layer config -> 对 Linear / MoE 回落到 unquantized method

对应源码在 vllm/model_executor/layers/quantization/humming.py:334-353。其中 Linear 路径会在 HummingLinearMethod.apply() 中展平输入并调用 HummingMethod.forward_layer(), 见 quantization/humming.py:610-623。MoE 路径则在 HummingMoEMethod.apply() 中调用 Humming experts, 见 quantization/humming.py:926-952。

GPT-OSS 特例:如果模型类型是 gpt_oss 且 HF quant method 是 mxfp4, 用户不能用 --quantization humming; 代码抛错提示改用 --moe-backend humming, 见 quantization/humming.py:212-225。

2. moe-backend=humming for MXFP4 / GPT-OSS

这是 Humming 作为 MoE backend 的路径,主要由 MXFP4 oracle 接入:

  • KernelConfig.moe_backend 接收 "humming", 见 vllm/config/kernel.py:122-137。
  • map_mxfp4_backend() 把 "humming" 映射为 [Mxfp4MoeBackend.HUMMING], 见 fused_moe/oracle/mxfp4.py:229-267。
  • backend_to_kernel_cls(HUMMING) 返回 BatchedHummingGroupedExperts、HummingGroupedExperts、HummingIndexedExperts, 见 fused_moe/oracle/mxfp4.py:112-171。
  • 显式 backend 优先于 auto: select_mxfp4_moe_backend() 在 runner_backend != "auto" 时只检查用户指定 backend, 见 fused_moe/oracle/mxfp4.py:404-460。

需要注意, auto priority 列表没有 Humming: GPT-OSS priority 包含 FlashInfer TRTLLM、AITER、Triton、FlashInfer CUTLASS、Marlin、XPU, 见 fused_moe/oracle/mxfp4.py:270-289; 通用 MXFP4 priority 包含 FlashInfer TRTLLM、DeepGEMM、Marlin, 见 fused_moe/oracle/mxfp4.py:292-311。因此 Humming 在 MXFP4 里是显式 --moe-backend humming 路径,不是 auto 默认候选。

三种 experts 的共同初始化会生成 compute config 和 tuning config:

  • use_f16_accum 来自 VLLM_HUMMING_USE_F16_ACCUM。
  • gemm_type 来自具体 experts 类的 humming_gemm_type()。
  • 分别为 w13 和 w2 调 HummingMethod.get_default_tuning_configs()。

对应源码在 fused_humming_moe.py:78-124。

Indexed 路径

HummingIndexedExperts.main_apply() 的关键步骤:

  1. 准备 workspace: prepare_buffers(), 见 fused_humming_moe.py:558-565。
  2. 用 moe_align_block_size() 生成 sorted_ids、expert_ids、num_tokens_padded, 见 fused_humming_moe.py:486-547。
  3. 调用 HummingMethod.may_quant_input(..., sublayer_name="w13"), 见 fused_humming_moe.py:573-578。
  4. 调用 HummingMethod.forward_layer(..., sublayer_name="w13") 做 gate/up GEMM, 见 fused_humming_moe.py:580-587。
  5. 调用 apply_activation() 做 MoE activation, 见 fused_humming_moe.py:589-593。
  6. 再次 may_quant_input(), 然后 forward_layer(..., sublayer_name="w2") 做 down GEMM, 见 fused_humming_moe.py:595-609。
  7. 用 moe_fused_mul_sum() 按 top-k 权重合并, 见 fused_humming_moe.py:611-617。

Grouped 路径

HummingGroupedExperts.main_apply() 先用 moe_permute() 按专家重排 token, 给 Humming grouped GEMM 提供 expert_layout=expert_first_token_offset; 两次 HummingMethod.forward_layer() 后, 用 moe_unpermute() 恢复 token 顺序并合并 top-k 权重。关键行号:

  • permute: fused_humming_moe.py:657-664
  • w13 Humming forward: fused_humming_moe.py:666-683
  • activation: fused_humming_moe.py:685-689
  • w2 Humming forward: fused_humming_moe.py:691-708
  • unpermute/combine: fused_humming_moe.py:710-716

Batched grouped 路径

BatchedHummingGroupedExperts 用 batched experts activation format, expert_layout 来自 expert_tokens_meta.expert_num_tokens。它的 w13/w2 调用同样落在 HummingMethod.forward_layer(), 见 fused_humming_moe.py:719-795。这个路径通常由 modular MoE 的 prepare/finalize 选择出来; HummingMoEMethod.select_gemm_impl() 在 activation format 为 BatchedExperts 时返回 BatchedHummingGroupedExperts, 见 quantization/humming.py:904-924。

与现有 kernel 的关系

Humming 不是 attention backend,也不是 sampling kernel。它和 Marlin、DeepGEMM、FlashInfer CUTLASS/TRTLLM、Triton、AITER 等处在同一类 “MoE expert computation backend” 选择空间里。MXFP4 oracle 的 backend 枚举把这些放在同一张表中, 见 fused_moe/oracle/mxfp4.py:60-89。

差异在于:

  • FlashInfer TRTLLM / CUTLASS 等路径通常直接选择对应外部 fused MoE kernel 类。
  • Marlin 路径有单独权重格式准备函数。
  • Humming 路径先通过 prepare_humming_moe_layer() 或 HummingMoEMethod.process_weights_after_loading() 转换成 Humming schema, 然后在 forward 时通过 HummingMethod.forward_layer() 做两段 GEMM。

如果用户没有显式选择 Humming, GPT-OSS/MXFP4 的 auto 列表不会默认探测 Humming。若要验证 Humming 性能,配置层应显式使用 --moe-backend humming, 并记录 VLLM_HUMMING_MOE_GEMM_TYPE、GPU capability、模型 quant config 和 profiler trace 中的实际外部 kernel 名称。

阅读结论

  • 确认: “humming kernel” 在 vLLM 中的具体对应是 HummingConfig + HummingLinearMethod/HummingMoEMethod + Humming*Experts 这一组适配实现,底层执行入口是外部 humming.layer.HummingMethod。
  • 确认: 它位于 MoE FFN/expert GEMM 环节,覆盖 prefill/decode 中所有会执行 MoE layer 的 token,不在 attention/KV/sampling 环节。
  • 确认: 集成机制包含 quantization method 注册、--moe-backend humming backend 映射、MXFP4 oracle 显式 backend 选择、权重 post-load 转换、MoE modular kernel prepare/finalize 编排。
  • 限制: vLLM 当前源码不能说明外部 humming 包内部 CUDA kernel 的精确 symbol; 需要 profiler 或外部包源码才能继续下钻。
