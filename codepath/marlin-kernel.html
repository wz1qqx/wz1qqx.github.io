vLLM Marlin kernel 集成学习笔记

结论先行:Marlin 在当前 vLLM 工作树里是一组自带 CUDA GEMM kernel 与 Python 适配层,主要服务量化 Linear GEMM 和 MoE expert GEMM。普通 Linear 最终落到 ops.marlin_gemm; MoE expert 先做 router/top-k,再用 ops.moe_wna16_marlin_gemm 分别执行 w13 gate/up GEMM 和 w2 down GEMM。

一句话定位

问题源码事实
Marlin kernel 是什么vLLM 自带的量化 GEMM 后端。CUDA 侧实现普通矩阵乘 marlin_gemm 和 MoE 专用 moe_wna16_marlin_gemm; Python 侧把 GPTQ/AWQ/FP8/MXFP8/MXFP4/NVFP4 权重重排成 Marlin 布局后调用这些 op。
对应实现文件核心 CUDA:csrc/quantization/marlin/* 与 csrc/moe/marlin_moe_wna16/*; Python op 封装:vllm/_custom_ops.py; Linear kernel:vllm/model_executor/kernels/linear/**/marlin.py; MoE experts:vllm/model_executor/layers/fused_moe/experts/marlin_moe.py; 量化适配:auto_gptq.py、awq_marlin.py、compressed_tensors_moe_wna16_marlin.py、marlin_utils*.py。
在推理流程的位置Linear GEMM 和 MoE FFN / expert GEMM。它不属于 attention、KV cache 或 sampling。prefill/decode 都会在执行量化 Linear 或 MoE FFN 时经过对应 Marlin 路径。
如何启用/选择Linear 侧通过 quantization override、--linear-backend marlin 或 auto priority 选择 Marlin kernel 类; MoE 侧通过 GPTQ/AWQ/CompressedTensors WNA16、FP8/MXFP4/NVFP4 oracle 的 --moe-backend marlin 或 auto priority 选择 MarlinExperts/BatchedMarlinExperts。
命名边界:marlin 不是 QuantizationMethods Literal 里的正式量化方法名;正式注册名是 gptq_marlin 和 awq_marlin, 见 vllm/model_executor/layers/quantization/__init__.py:12-47。不过 GPTQ/AWQ override 会接受用户传入 quantization="marlin" 作为触发条件,见 auto_gptq.py:216-236 与 awq_marlin.py:248-269。Kernel backend 的正式选项则包含 --linear-backend marlin 和 --moe-backend marlin,见 vllm/config/kernel.py:122-209。

推理流程位置

Marlin 不是单一功能开关,而是两个推理阶段会用到的 GEMM 后端。

路径运行位置prefill/decode 关系关键入口
量化 Linearattention 的 q/k/v/o projection、MLP dense projection、LM head 等所有被量化配置命中的 LinearBase。prefill 与 decode 都执行 transformer layer 的 Linear,因此只要对应 layer 的 quant method 选到 Marlin,两个阶段都会走。AutoGPTQLinearMethod.apply() / AWQMarlinLinearMethod.apply() 调 kernel apply_weights(),见 auto_gptq.py:451-460 与 awq_marlin.py:491-507。
MoE expert GEMMMoE FFN 中 router 选专家之后的 expert computation。先做 w13 gate/up GEMM,中间做 activation,再做 w2 down GEMM。prefill/decode 都会在 MoE layer forward 时使用同一路径;差异主要体现在 token 数 M、routing 分布、CUDA graph/EP 通信,不是 Marlin 调用链的不同。FusedMoE.forward() 到 MoERunner._apply_quant_method(),再进入 MarlinExperts.apply() 或旧式 fused_marlin_moe(),见 fused_moe/layer.py:1306-1316、runner/moe_runner.py:496-537。

vLLM 的 modular MoE 注释把整个 MoE 分成:

[Router] -> [Quantize-Dispatch] -> [Permute-Experts-Unpermute] -> [Combine]

这段分层定义见 vllm/model_executor/layers/fused_moe/modular_kernel.py:46-80。Marlin expert 属于 FusedMoEExpertsModular 的 “matmul + activation + matmul” 部分; prepare/finalize 可以由其它 all2all/dispatch 实现组合。

Linear 集成路径

1. 注册与 backend 选择

  • QuantizationMethods 包含 "gptq_marlin" 和 "awq_marlin",见 quantization/__init__.py:12-47。
  • get_quantization_config() 把 "gptq_marlin" 映射到 AutoGPTQConfig,把 "awq_marlin" 映射到 AWQMarlinConfig,见 quantization/__init__.py:109-170。
  • ModelConfig._verify_quantization() 会按 override 列表探测模型 HF quant config,其中包含 gptq_marlin、awq_marlin、moe_wna16 等,见 vllm/config/model.py:1002-1044; 探测结果会写回 self.quantization,见 vllm/config/model.py:1046-1068。
  • --linear-backend marlin 会把 Linear 候选 kernel 过滤到 MarlinFP8ScaledMMLinearKernel、MarlinLinearKernel、MarlinMxfp8LinearKernel、MarlinNvFp4LinearKernel、MarlinMxFp4LinearKernel,见 vllm/model_executor/kernels/linear/__init__.py:176-207。

2. GPTQ / AWQ WNA16 Linear

GPTQ 与 AWQ 都先构造 MPLinearLayerConfig,再通过 choose_mp_linear_kernel() 选择混合精度 kernel。CUDA 平台的 auto priority 中,Marlin 排在 Cutlass W4A8、Machete、AllSpark 之后,Conch/Exllama 之前,见 vllm/model_executor/kernels/linear/__init__.py:331-339。

  • AutoGPTQLinearMethod.create_weights() 构造 weight/scales/qzeros/g_idx 参数并调用 choose_mp_linear_kernel(),见 auto_gptq.py:322-350。
  • AWQMarlinLinearMethod.create_weights() 同样构造 MPLinearLayerConfig 并选择 kernel,见 awq_marlin.py:402-489。
  • MarlinLinearKernel.can_implement() 要求 CUDA、支持的 weight type、group size 与 shape,见 kernels/linear/mixed_precision/marlin.py:30-62。
  • MarlinLinearKernel.process_weights_after_loading() 分配 workspace、处理 act-order、调用 ops.gptq_marlin_repack()、重排 scales/zero-points/bias,见 kernels/linear/mixed_precision/marlin.py:67-171。
  • MarlinLinearKernel.apply_weights() 调 apply_gptq_marlin_linear(),见 kernels/linear/mixed_precision/marlin.py:173-200。
  • apply_gptq_marlin_linear() 可能先对 activation 做 INT8/FP8 per-token quant,最后调用 ops.marlin_gemm(),见 quantization/utils/marlin_utils.py:510-574。

MoE 集成路径

1. FusedMoE 到 MarlinExperts

MoE layer 构造时会保存 kernel_config.moe_backend,再通过量化配置拿到 quant method:

  • FusedMoE.__init__ 构造 FusedMoEConfig 并记录 moe_backend=vllm_config.kernel_config.moe_backend,见 fused_moe/layer.py:318-341。
  • 同一个构造流程调用 quant_config.get_quant_method(self, prefix),如果没有 quant method 就回落到 UnquantizedFusedMoEMethod,见 fused_moe/layer.py:351-369。
  • 权重加载结束后,模型加载器遍历模块并调用每个 quant method 的 process_weights_after_loading(),见 model_loader/utils.py:99-112。
  • FusedMoE.forward() 进入 runner,见 fused_moe/layer.py:1306-1316。
  • MoERunner.forward() 通过 custom op 入口回到 _forward_impl(),见 runner/moe_runner.py:592-642 与 runner/moe_runner.py:90-105。
  • _forward_impl() 确保 quant config 初始化、可选 gate、dispatch,再调用 _apply_quant_method(),见 runner/moe_runner.py:742-799。
  • _apply_quant_method() 对非 monolithic 路径先 router.select_experts(),拿到 topk_weights/topk_ids,再调用 quant method 的 apply(),见 runner/moe_runner.py:496-537。

2. Marlin MoE expert 具体做什么

vllm/model_executor/layers/fused_moe/experts/marlin_moe.py 是 MoE Marlin 的核心 Python 适配。它有两种入口:

入口activation format来源最终 GEMM
MarlinExpertsStandard普通 token/top-k 格式。MarlinExperts.apply() 调 fused_marlin_moe(),见 marlin_moe.py:693-797。两次 ops.moe_wna16_marlin_gemm()。
BatchedMarlinExpertsBatchedExpertsexpert-parallel/dispatch 后已经按专家 batch 化的格式。batched_fused_marlin_moe() 注释说明把 [B, MAX_TOKENS, K] 视作 2D contiguous tensor,见 marlin_moe.py:390-446。仍然通过 _fused_marlin_moe() 和 ops.moe_wna16_marlin_gemm()。

_fused_marlin_moe() 的关键执行顺序:

  1. 计算 M,K,N,准备 workspace 和两个 intermediate cache,见 marlin_moe.py:96-124。
  2. 如配置了 VLLM_MARLIN_INPUT_DTYPE,对 hidden states 做 INT8/FP8 activation quant,见 marlin_moe.py:125-133。
  3. 第一次 ops.moe_wna16_marlin_gemm() 计算 w13 gate/up projection,见 marlin_moe.py:134-161。
  4. 调用 apply_moe_activation() 或传入的 activation callback,见 marlin_moe.py:162-173。
  5. 可选对 activation 后的 intermediate 再量化,见 marlin_moe.py:181-191。
  6. 第二次 ops.moe_wna16_marlin_gemm() 计算 w2 down projection,并根据 apply_router_weight_on_input 决定 top-k 权重乘法位置,见 marlin_moe.py:193-220。
  7. fused_marlin_moe() 把输出 view 成 [M, topk, K],最后 torch.sum(..., dim=1) 或自定义 moe_sum 合并 top-k,见 marlin_moe.py:342-387。

3. WNA16 MoE: GPTQ / AWQ / CompressedTensors

WNA16 这里指 weight-only int4/int8 + 16-bit activation 的 MoE expert GEMM。GPTQ/AWQ 的集成从 quant method 到 modular kernel 已经迁移到 oracle:

  • AutoGPTQConfig.get_quant_method() 对 RoutedExperts 返回 AutoGPTQMoEMethod,前提是 check_moe_marlin_supports_layer() 通过,见 auto_gptq.py:238-258。
  • AWQMarlinConfig.get_quant_method() 对 RoutedExperts 返回 AWQMarlinMoEMethod,不支持则回落到 MoeWNA16Config,见 awq_marlin.py:280-324。
  • AutoGPTQMoEMethod 与 AWQMarlinMoEMethod 都在 __init__ 里调用 select_wna16_moe_backend(),见 auto_gptq.py:463-487 与 awq_marlin.py:510-525。
  • select_wna16_moe_backend() 的候选只有 MARLIN 和 BATCHED_MARLIN,并通过 is_supported_config() 按 activation format 选择,见 fused_moe/oracle/int_wna16.py:37-145。
  • make_wna16_moe_kernel() 用 maybe_make_prepare_finalize() 创建 dispatch/finalize,再实例化 MarlinExperts 或 BatchedMarlinExperts,最后返回 FusedMoEKernel,见 fused_moe/oracle/int_wna16.py:148-206。

权重转换同样在 oracle 中集中完成:

  • convert_to_wna16_moe_kernel_format() 根据 backend 分发到 GPTQ 或 AWQ Marlin 后处理,见 fused_moe/oracle/int_wna16.py:528-620。
  • GPTQ 路径排序/清空 g_idx,调用 ops.gptq_marlin_moe_repack() 处理 w13/w2,并重排 scales/bias,见 fused_moe/oracle/int_wna16.py:214-382。
  • AWQ 路径调用 ops.awq_marlin_moe_repack(),并用 moe_awq_to_marlin_zero_points() 转换 zero point,见 fused_moe/oracle/int_wna16.py:385-525。

CompressedTensors 的 WNA16 MoE 也优先使用 Marlin:

  • 如果 check_moe_marlin_supports_layer() 通过且不是 ROCm,返回 CompressedTensorsWNA16MarlinMoEMethod,见 compressed_tensors_moe.py:96-125。
  • 该方法创建 Marlin 布局的 w13_weight_packed/w2_weight_packed 与 scale/g_idx 参数,见 compressed_tensors_moe_wna16_marlin.py:161-323。
  • 加载后处理调用 ops.gptq_marlin_moe_repack()、marlin_moe_permute_scales() 并创建 workspace,见 compressed_tensors_moe_wna16_marlin.py:324-459。
  • 标准 modular 路径通过 select_gemm_impl() 返回 MarlinExperts/BatchedMarlinExperts,见 compressed_tensors_moe_wna16_marlin.py:474-510; 旧式 apply 路径直接调用 fused_marlin_moe(),见 compressed_tensors_moe_wna16_marlin.py:542-577。

与其它 kernel 的关系

Marlin 在 vLLM 中不是独立模型流程,而是多个 quantization family 下的 backend 候选。它与 Cutlass、FlashInfer、DeepGEMM、Triton、Machete、Conch、Exllama、AITER 等通过同一套 selector/oracle 竞争。

  • Linear WNA16: choose_mp_linear_kernel() 按平台 priority 和 --linear-backend 过滤选择,见 kernels/linear/__init__.py:611-681。
  • Scaled MM/FP8: choose_scaled_mm_linear_kernel() 负责按 possible kernels、force kernel 和 --linear-backend 过滤,见 kernels/linear/__init__.py:430-499。
  • MXFP4/NVFP4 Linear: init_mxfp4_linear_kernel() 与 init_nvfp4_linear_kernel() 负责处理 auto、显式 backend 和 deprecated env,见 kernels/linear/__init__.py:730-781 与 kernels/linear/__init__.py:831-946。
  • MoE: 各 quantization oracle 把 --moe-backend marlin 映射到 enum,再映射到 MarlinExperts/BatchedMarlinExperts。FP8/MXFP4/NVFP4 的 mapping 分别见 oracle/fp8.py:204-220、oracle/mxfp4.py:229-267、oracle/nvfp4.py:141-157。
  • Modular MoE: FusedMoEKernel 要求 prepare/finalize 与 experts 都是 modular 或都 monolithic,并在 apply() 中串起 prepare、experts、finalize,见 modular_kernel.py:1507-1543 与 modular_kernel.py:1633-1660。

阅读结论

  • 确认: Marlin 在 vLLM 里是自带 CUDA op + Python selector/quant method 适配,不是只依赖外部包。
  • 确认: 普通量化 Linear 的最终核心 op 是 ops.marlin_gemm; MoE expert 的最终核心 op 是 ops.moe_wna16_marlin_gemm。
  • 确认: 它在模型推理中位于 GEMM 环节:attention/MLP/LM head 的量化 Linear GEMM,以及 MoE FFN 的 routed expert GEMM。prefill 和 decode 都可能经过它。
  • 确认: 集成机制包括 quantization override、linear backend selector、MoE backend oracle、post-load 权重重排、modular MoE prepare/finalize 组合。
  • 注意: --quantization marlin 在源码中表现为 GPTQ/AWQ override 可接受的别名触发条件,不是 QuantizationMethods 的正式注册名;文档和实验记录应写清最终解析后的 self.quantization。
