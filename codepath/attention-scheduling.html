vLLM Attention 调度与 split-kv 学习笔记

结论先行:vLLM v1 的 attention 相关“调度”分成两层。全局 Scheduler 只决定本步哪些 request 跑多少 token,并用 token budget、chunked prefill、running-first/waiting-next 来混排长短请求;它不直接生成 attention tile 计划。真正的 split-kv、tile scheduler metadata、prefill/decode 拆分发生在 attention backend 的 metadata builder 和 kernel wrapper 里,例如 FlashAttention-3 的 scheduler_metadata、FlashMLA 的 tile_scheduler_metadata/num_splits、Triton MLA decode 的 num_kv_splits。

一句话定位

问题vLLM 源码事实和 HPC-Ops 题述方案的关系
全局 scheduler 是否有 prefill/decode 两个阶段没有。Scheduler.schedule() 注释明确说 scheduler 内没有 decoding phase / prefill phase,只有每个 request 的 num_computed_tokens 和 num_tokens_with_spec,每步让前者追上后者。见 vllm/v1/core/sched/scheduler.py:335-345。vLLM 的长短混排首先是 request/token-budget 粒度,不是 attention tile 粒度的全局负载均衡。

全局 scheduler 的输出字段是 request/token 级别的 SchedulerOutput:新请求、已缓存请求、num_scheduled_tokens、总 token 数、spec decode token、encoder input 和 common prefix blocks。见 vllm/v1/core/sched/output.py:180-207。这些字段随后被 GPU runner 转成 attention 需要的连续 token 布局。

Scheduler: token budget 与 chunked prefill

Scheduler.__init__ 把 max_num_scheduled_tokens 设置为 config 显式值,否则回退到 max_num_batched_tokens,见 scheduler.py:106-112。每步 scheduling 从 running request 开始,然后处理 waiting request。running request 的新增 token 量会被 long_prefill_token_threshold 和当前 token budget 裁剪,见 scheduler.py:370-398。waiting request 在命中本地/外部 prefix cache 后计算待跑 token;如果 chunked prefill 没开且请求超过 budget,会停止排队而不是切块,见 scheduler.py:596-675。

# vllm/v1/core/sched/scheduler.py:335-345
def schedule(self) -> SchedulerOutput:
    # NOTE(woosuk) on the scheduling algorithm:
    # There's no "decoding phase" nor "prefill phase" in the scheduler.
    # Each request just has the num_computed_tokens and
    # num_tokens_with_spec. num_tokens_with_spec =
    # len(prompt_token_ids) + len(output_token_ids) + len(spec_token_ids).
    # At each step, the scheduler tries to assign tokens to the requests
    # so that each request's num_computed_tokens can catch up its
    # num_tokens_with_spec. This is general enough to cover
    # chunked prefills, prefix caching, speculative decoding,

这解释了 vLLM “长短请求混排”的第一层含义:长 prefill 不一定一次占满整步,它会被 token budget 和 threshold 切成多个 step;decode 或较短请求也可以继续被调度。但这里没有 attention tile、SM、split-kv 的计划。

GPU runner: 从 SchedulerOutput 到 CommonAttentionMetadata

GPU runner 先按 SchedulerOutput.num_scheduled_tokens 构造 token 序列和每个 request 的 cumulative query length。关键字段:

  • query_start_loc:每个 request 在 query tensor 中的起点,见 gpu_model_runner.py:1976-1983。
  • seq_lens:num_computed_tokens + num_scheduled_tokens,见 gpu_model_runner.py:2088-2091。
  • slot_mapping:由 block table 根据 position 计算 KV cache 写入位置,见 gpu_model_runner.py:2093-2097。
  • is_prefilling:num_computed_tokens_cpu < num_prompt_tokens_cpu,供部分 backend 区分真实 decode 和 short extend,见 gpu_model_runner.py:2267-2273。
# vllm/v1/worker/gpu_model_runner.py:2280-2296
cm_base = CommonAttentionMetadata(
    query_start_loc=self.query_start_loc.gpu[: num_reqs_padded + 1],
    query_start_loc_cpu=self.query_start_loc.cpu[: num_reqs_padded + 1],
    seq_lens=self.seq_lens[:num_reqs_padded],
    _seq_lens_cpu=seq_lens_cpu,
    _num_computed_tokens_cpu=num_computed_tokens_cpu,
    seq_lens_cpu_upper_bound=seq_lens_cpu_upper_bound,
    num_reqs=num_reqs_padded,
    num_actual_tokens=num_tokens_padded,
    max_query_len=max_query_len,
    max_seq_len=max_seq_len,
    block_table_tensor=block_table_gid_0,
    slot_mapping=slot_mapping_gid_0,
    causal=True,
    is_prefilling=is_prefilling,
    positions=self.positions[:num_tokens_padded],
)

CommonAttentionMetadata 是 backend 间的共同输入。它定义 query_start_loc、seq_lens、max_query_len、max_seq_len、block table、slot mapping、is_prefilling 和 seq_lens_cpu_upper_bound;并提供 compute_num_computed_tokens() 用 seq_lens - query_lens 在 device 上恢复上下文长度。见 vllm/v1/attention/backend.py:352-410 与 backend.py:457-462。

GPU runner 有一个 backend-driven reorder hook。注释说明部分 attention backend,尤其 MLA,希望按 attention 计算是 compute-bound 还是 memory-bound 来分离 request,见 vllm/v1/worker/gpu_model_runner.py:1054-1077。真正的 reorder helper 把 batch 分成四段:

decode:        num_scheduled <= threshold 且不是 prefilling
short_extend:  num_scheduled <= threshold 且仍在 chunked prefill
long_extend:   num_scheduled > threshold 且仍在 chunked prefill
prefill:       num_computed == 0

这段定义见 vllm/v1/attention/backends/utils.py:632-646。注意这里的 reorder 是为了 metadata/backend 路径判断,不是 scheduler 自身维护 phase。

backend 再调用 split_decodes_and_prefills()。这个 helper 假设 batch 已重排,用 query_lens、decode_threshold、是否要求 uniform query length、是否把 short extend 当 decode 来决定边界。见 vllm/v1/attention/backends/utils.py:533-602。

# vllm/model_executor/layers/attention/mla_attention.py:1709-1715
num_decodes, num_prefills, num_decode_tokens, num_prefill_tokens = (
    split_decodes_and_prefills(
        common_attn_metadata,
        decode_threshold=self.reorder_batch_threshold,
        require_uniform=(self.query_len_support != QueryLenSupport.VARLEN),
    )
)

MLA 的文件头说明了为什么要拆两条路径:MLA 一般希望 prefill 走 compute-friendly approach,decode 走 data-movement-friendly approach;当前“小/大”的划分由 scheduler 标出的 prefill/decode 标签间接决定,并且源码注释说这个阈值未来应该调优,见 mla_attention.py:13-21。MLA builder 在 num_prefills > 0 时构造 MLACommonPrefillMetadata 和 chunked context metadata,在 num_decodes > 0 时调用 _build_decode(),最后组合成 MLACommonMetadata,见 mla_attention.py:1760-1937 与 mla_attention.py:1938-1999。

split-kv 与 scheduler_metadata

FlashAttention-3: AOT scheduler metadata

FlashAttentionMetadata 包含 scheduler_metadata、prefix_scheduler_metadata 和 max_num_splits,见 vllm/v1/attention/backends/flash_attn.py:222-255。builder 初始化时只有 FA3 会启用 AOT schedule;full CUDA graph + FA3 下预分配 scheduler_metadata,注释列出 FA3 scheduler metadata 中用于 prepare_varlen、dynamic_split、sort_batches、head_swizzle 的 slots,见 flash_attn.py:332-374。

get_scheduler_metadata() 只用于 FA3,最终调用 torch.ops._vllm_fa3_C.get_scheduler_metadata;其 num_splits 注释为 “Can be tuned for speed”,见 vllm/vllm_flash_attn/flash_attn_interface.py:121-173。FA2 wrapper 对包含 scheduler_metadata 与 descale 参数的组合有限制,且 num_splits > 1 会抛 NotImplementedError;FA3 则把 scheduler_metadata 和 num_splits 传给 torch.ops._vllm_fa3_C.fwd,见 flash_attn_interface.py:285-366。

FlashMLA: tile scheduler metadata 与 num_splits

FlashMLA decode metadata 继承 MLACommonDecodeMetadata,额外带 scheduler_metadata,见 vllm/v1/attention/backends/mla/flashmla.py:99-106。它的 builder 设置 query_len_support=UNIFORM 且 reorder_batch_threshold=128,注释说明会把 small prefill 放到 decode pathway 处理,见 flashmla.py:109-113。

# vllm/v1/attention/backends/mla/flashmla.py:152-201
def _build_decode(...):
    query_lens_cpu = query_start_loc_cpu[1:] - query_start_loc_cpu[:-1]
    max_query_len = query_lens_cpu.max().item()
    num_q_tokens_per_head_k = max_query_len * self.num_q_heads // 1
    scheduler_metadata, _ = get_mla_metadata(
        seq_lens_device,
        num_q_tokens_per_head_k,
        1,  # MQA for the decode path
        is_fp8_kvcache=self.is_fp8_kvcache,
    )
    if self.is_fp8_kvcache:
        tile_scheduler_metadata, num_splits = get_mla_metadata_dense_fp8(...)
        ...
        scheduler_metadata.tile_scheduler_metadata = tile_scheduler_metadata
        scheduler_metadata.num_splits = num_splits

    return FlashMLADecodeMetadata(..., scheduler_metadata=scheduler_metadata, ...)

非 FP8 路径的 get_mla_metadata 来自 FlashMLA 第三方接口;FP8 dense 路径通过 torch.ops._flashmla_extension_C.get_mla_decoding_metadata_dense_fp8 返回 tile_scheduler_metadata 和 num_splits,见 vllm/v1/attention/ops/flashmla.py:86-120。另一个 custom op wrapper 的 docstring 也说明返回值形状为 tile_scheduler_metadata: (num_sm_parts, TileSchedulerMetaDataSize) 和 num_splits: (batch_size + 1),见 vllm/_custom_ops.py:2988-3000。

forward 阶段,FlashMLA decode 从 metadata 取 scheduler metadata。FP8 KV cache 路径把 tile_scheduler_metadata 和 num_splits 传给 flash_mla_with_kvcache_fp8;非 FP8 路径把整个 scheduler metadata 传给 flash_mla_with_kvcache,见 flashmla.py:274-330。FP8 op wrapper 最终把这两个 tensor 传给 torch.ops._flashmla_extension_C.fwd_kvcache_mla_fp8,见 ops/flashmla.py:123-152;普通 custom op wrapper 则传给 torch.ops._C.flash_mla_fwd_kvcache,见 _custom_ops.py:3003-3044。

Triton MLA: Python 端全 batch num_kv_splits

Triton MLA decode 路径没有类似 FlashMLA 的 per-request num_splits tensor。它在 Python forward 中依据 batch 的 max_seq_len 算一个 num_kv_splits:batch invariant 强制 1;否则先按 max_seq_len // 512 取理想 split,再取 2 的幂,并用 SM count * 2 封顶。见 vllm/v1/attention/backends/mla/triton_mla.py:157-175。

# vllm/v1/attention/backends/mla/triton_mla.py:157-175
if envs.VLLM_BATCH_INVARIANT:
    num_kv_splits = 1
else:
    min_work_per_split = 512
    ideal_splits = max(1, attn_metadata.max_seq_len // min_work_per_split)
    ideal_splits = triton.next_power_of_2(ideal_splits)
    occupancy_multiplier = 2
    max_splits = self._sm_count * occupancy_multiplier
    num_kv_splits = min(ideal_splits, max_splits)

对应 Triton kernel 的 stage1 用 tl.program_id(2) 作为 split_kv_id,把当前 request 的 KV 长度均分成 NUM_KV_SPLITS 段,只遍历本段 KV,并把局部输出和 LSE 写到中间 buffer。见 vllm/v1/attention/ops/triton_decode_attention.py:90-192。stage2 再遍历所有 split,用 LSE 合并局部 softmax 输出,写出最终 o 和 lse,见 triton_decode_attention.py:560-612。调用链在 decode_attention_fwd_normal() / decode_attention_fwd_grouped() 中串起 stage1 和 stage2,见 triton_decode_attention.py:657-689 与 triton_decode_attention.py:692-726。

指定模块阅读

chunked_local_attention: local window 的 virtual batch

ChunkedLocalAttention 不是全局 scheduler 的 chunked prefill,也不是 split-kv。它把底层 attention backend 包一层,在 metadata build 前调用 make_local_attention_virtual_batches(),把每个原始序列按 local attention chunk 切成多个“virtual batch item”,再交给普通 flash attention backend。见 vllm/model_executor/layers/attention/chunked_local_attention.py:30-78 与 chunked_local_attention.py:81-129。

make_local_attention_virtual_batches() 的注释举例说明:原本一个 batch item 的 local attention mask 可以通过拆成多个 local batch item 来模拟,每个 local batch item 有自己的 q length、kv length 和 local block table。见 vllm/v1/attention/backends/utils.py:195-247。实现上它根据 attn_chunk_size、query_start_loc_cpu、seq_lens_cpu 计算 local blocks、local cu_seqlens_q、seqlens_k_local、local block table,最后返回新的 CommonAttentionMetadata,见 utils.py:271-390。

mla_attention: decode/prefill 双路径和 chunked context

MLACommonMetadata 明确记录 num_decodes、num_decode_tokens、num_prefills、num_incoming_decodes,并持有 prefill、decode、incoming_decode_metadata 三类子 metadata,见 vllm/model_executor/layers/attention/mla_attention.py:1352-1389。prefill 子 metadata 可包含 chunked context metadata,包括 cu_seq_lens、chunk start、每 chunk 总 token、workspace、token-to-seq 映射等,见 mla_attention.py:1307-1340。

MLA 对 chunked prefill 的处理是 context workspace 分块:当 prefill 有历史 context 时,它把每个 prefill 的 context 按 workspace 上限切 chunk,当前实现给每个带 context 的 prefill 分配相等 workspace,源码注释也说可以用更高级算法给长 context 更多 workspace。见 mla_attention.py:1761-1784。这和 HPC-Ops 题述的“Tile 均衡”方向相关,但源码中这里还不是动态 tile 均衡策略,而是 prefill context gather/compute 的 chunk metadata。

cross_attention: encoder-decoder metadata 改写

CrossAttentionBuilder 复用底层 backend,但 build 时会复制 common metadata 并改成 cross-attention 语义:设置 causal=False,把 max_seq_len 改为 encoder 最大长度;用 seq_lens_cpu_upper_bound - query_lens 判断已有 cache 的 decode row;对这些 decode row 把 encoder_seq_lens_cpu 置 0,避免重复 reshape/cache;然后把 seq_lens 改成 encoder seq lens,并用 cross KV cache 的 block table 重新计算 slot mapping。见 vllm/model_executor/layers/attention/cross_attention.py:81-131。

CrossAttention 选择 underlying backend 时显式传 attn_type=AttentionType.ENCODER_DECODER,并返回 CrossAttentionSpec 作为 KV cache spec,见 cross_attention.py:216-240。这条路径主要是 encoder-decoder KV cache 语义调整,不是 split-kv 调度。

DBO / ubatching: 执行切片不是 tile scheduling

DBO 配置位于 ParallelConfig:enable_dbo、ubatch_size、decode 阈值 32、prefill 阈值 512,见 vllm/config/parallel.py:196-210。check_ubatch_thresholds() 根据 batch 是否 uniform decode 选择 decode/prefill 阈值,见 vllm/v1/worker/ubatch_utils.py:38-46。

maybe_create_ubatch_slices() 按 cumulative scheduled tokens 计算 token split points,并把每个 token slice 映射到 request slice,见 ubatch_utils.py:63-114。_make_metadata_with_slice() 会修正被切开的首尾 request 的 query_start_loc、seq_lens、block table、slot mapping 等,生成每个 ubatch 的 CommonAttentionMetadata,见 ubatch_utils.py:134-248。GPU runner 在 execute path 中创建 ubatch slices,再传给 _build_attention_metadata() 和 set_forward_context(),见 gpu_model_runner.py:4082-4088 与 gpu_model_runner.py:4166-4224。

所以 DBO 和 HPC-Ops 题述 “长短请求混排治长尾”有目标相近的一面:都在避免大工作量阻塞小工作量。但 DBO 是模型执行 microbatch 粒度,不是 attention KV split 或 tile-level load balancer。

split-kv 决策题述强调 split-kv。存在多个 backend-local split-kv 策略:FA3 可用内部 heuristic/AOT scheduler;FlashMLA 由 metadata op 生成 num_splits;Triton MLA 用全 batch max_seq_len 算固定 split 数。flash_attn.py:430-468, flashmla.py:152-201, triton_mla.py:157-175。
Tile 均衡题述强调 Tile 均衡。FA3 scheduler metadata 注释包含 dynamic_split/sort_batches/head_swizzle;FlashMLA metadata 包含 tile scheduler metadata。vLLM Python 层没有统一 tile scheduler,更多是委托 FA3/FlashMLA custom op。flash_attn.py:354-359, _custom_ops.py:2994-2997。具体 tile 分配算法在 C++/外部 op 内,本文不臆造。
混合长短请求题述通过混排减少长尾。vLLM 有三种相关机制:global token budget/chunked prefill;backend reorder decode/short_extend/long_extend/prefill;DBO ubatch 按 token slice 拆执行。scheduler.py:335-398, utils.py:632-646, ubatch_utils.py:63-114。三者粒度不同。
统一性题述听起来像一个 attention 调度方案。vLLM 是 backend 多态:FlashAttention、FlashMLA、Triton MLA、chunked local、cross attention 的 metadata build 都不同。gpu_model_runner.py:2328-2397 按每个 attention group 的 builder build metadata。
common prefix / cascade题述未提。vLLM FlashAttention 有 cascade attention,利用 common prefix blocks 优化共享前缀,但这是 common-prefix 优化,不等同于长尾 split-kv 调度。SchedulerOutput.num_common_prefix_blocks 见 output.py:205-207;FA build 中 prefix/suffix schedule 见 flash_attn.py:510-534。

阅读结论

  • 确认: vLLM 现状不是一个统一的 “Attention 动态负载调度器”,而是全局 token scheduler + per-backend attention metadata/kernel scheduler 的组合。
  • 确认: vLLM 已经有多处和 HPC-Ops 题述方向相似的机制:chunked prefill 降低长请求 head-of-line blocking;MLA/FlashMLA 按 decode/prefill 走不同路径;FA3/FlashMLA/Triton MLA 都存在 split 或 tile metadata。
  • 边界: vLLM 的 split-kv 决策不是一个全局统一策略。FA3 的动态 split/sort/head swizzle 在 FA3 op metadata 内,FlashMLA 的 tile scheduler 在 FlashMLA metadata op 内,Triton MLA 是 Python 端按 batch 最大长度给全 batch 一个 split 数。
  • 边界: DBO/ubatching 和 chunked local attention 容易被误读。前者是 model executor microbatch 切片,后者是 local attention virtual batch 改写;两者都不是 HPC-Ops 题述意义上的 attention tile load balancing。
  • 对照结论: 如果把 HPC-Ops 看作“在 attention 层动态拆 KV、均衡 tile、混排长短请求以治长尾”的方案,那么 vLLM 当前更像是把同类问题分散到 scheduler、runner reorder、DBO、backend metadata builder 和具体 kernel 中分别处理。相同点是都有 split 和混排;不同点是 vLLM 的策略边界更 backend-local,Python 层能看到的是 metadata 和阈值,不是统一 tile 调度算法。
