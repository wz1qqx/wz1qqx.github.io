vLLM MoE router / gating 精度与选择流程学习笔记

结论先行:当前工作树里的 MoE router/gating 被拆成两层:先由模型侧 gate、GateLinear 或 NormGateLinear 生成 router_logits,再由 create_fused_moe_router() 选出的 router 把 logits 变成 topk_weights 和 topk_ids。vLLM 已经在多个精度敏感路径上把 router logits 或 top-k weights 保持为 FP32,但本地源码没有看到 HPC-Ops 题述那种“双 BF16 模拟 FP32 Router GEMM”的实现。

公共 select 模板

所有继承 BaseRouter 的 router 都共享同一个 select_experts() 模板。子类只实现 _compute_routing();公共模板负责 EPLB 状态校验、logical id 捕获、EPLB logical-to-physical 映射和 indices dtype 转换。

# vllm/model_executor/layers/fused_moe/router/base_router.py:250-298
def select_experts(self, hidden_states, router_logits, *, input_ids=None):
    self._validate_eplb_state()
    indices_type = self._get_indices_type()
    topk_weights, topk_ids = self._compute_routing(
        hidden_states, router_logits, indices_type, input_ids=input_ids
    )
    if self.capture_fn is not None:
        self.capture_fn(topk_ids)
    topk_ids = self._apply_eplb_mapping(topk_ids)
    topk_ids = self._convert_indices_dtype(topk_ids, indices_type)
    return topk_weights, topk_ids

EPLB 映射在 CUDA-like 平台上用 Triton kernel 同时完成 logical expert 到 physical expert 的映射和 load 计数记录,见 base_router.py:17-125。这意味着普通 router 输出的是 logical expert id;如果启用 EPLB,进入 expert backend 前会被转换成 physical expert id。

默认 fused top-k

FusedTopKRouter 是最普通的 top-k 路径。它先按 token 数 M 分配三个 buffer:

  • topk_weights: [M, topk], dtype 固定为 torch.float32。
  • topk_ids: [M, topk], dtype 是 modular kernel 要求的 indices dtype,否则 int32。
  • token_expert_indices: [M, topk], int32,用于底层 op 的 token-expert 元数据。

随后按 scoring_func 选择 softmax 或 sigmoid kernel。ROCm AITER enabled 时会切到 AITER top-k op,否则用 vLLM 自带 ops.topk_softmax/ops.topk_sigmoid。见 fused_topk_router.py:69-113。

# vllm/model_executor/layers/fused_moe/router/fused_topk_router.py:81-113
topk_weights = torch.empty(M, topk, dtype=torch.float32, device=hidden_states.device)
topk_ids = torch.empty(M, topk, dtype=torch.int32 if indices_type is None else indices_type,
                       device=hidden_states.device)
token_expert_indices = torch.empty(M, topk, dtype=torch.int32, device=hidden_states.device)

if scoring_func == "softmax":
    topk_func = dispatch_topk_softmax_func(...)
elif scoring_func == "sigmoid":
    topk_func = dispatch_topk_sigmoid_func(...)
else:
    raise ValueError(...)
topk_weights, topk_ids = topk_func(..., gating_output, renormalize)

CUDA 侧 topk_softmax()/topk_sigmoid() 支持 gating_output 为 FP32/FP16/BF16,workspace 用 FP32,输出写入 Python 侧传入的 FP32 topk_weights。见 csrc/moe/topk_softmax_kernels.cu:791-865。

Tier触发条件输出精度源码依据

CUDA 侧 dsv3_router_gemm 的约束很窄:输入和权重必须 BF16,hidden dim 固定 7168,专家数只能是 256 或 384,token 数 1 到 16,SM 要在 90 到 103,输出只能 FP32 或 BF16。见 csrc/moe/moe_ops.h:69-77, csrc/moe/dsv3_router_gemm_entry.cu:101-130。这与 HPC-Ops 题述方案的共同点是都重视 router GEMM 的精度;差异是 vLLM 这里是 BF16 GEMM 产生 FP32/BF16 输出,源码没有双 BF16 模拟 FP32 累加的代码证据。

NormGateLinear: DSV4 的 fused norm + gate

NormGateLinear 是 DeepSeek V4 MoE routing 的专用层,同时持有 RMSNorm 和 GateLinear。它的语义是返回 (normed_x, logits),其中 logits 固定 FP32。

  • DSV4-Pro shape 命中时,调用 custom op torch.ops.vllm.dsv4_pro_norm_gate,底层是 dsv4_norm_router_gemm,融合 RMSNorm 和 router GEMV。见 router/norm_gate_linear.py:19-36, router/norm_gate_linear.py:91-108。

RoutingMethodType 与 backend 支持

router 创建后会把 routing_method_type 写进 FusedMoEConfig,这不是单纯的标签,后续 expert backend 会用它判断能否支持当前 routing。映射规则在 get_routing_method_type():

  • sqrtsoftplus + renormalize 对应 DeepseekV4。
  • 有 bias 且 grouped sigmoid 对应 DeepSeekV3;有 bias 且普通 sigmoid 对应 MiniMax2。
  • 无 bias sigmoid top-1 对应 Llama4;无 bias sigmoid + renormalize 对应 SigmoidRenorm。
  • softmax + renormalize 对应 RenormalizeNaive;softmax 不 renormalize 对应 Default。

源码见 fused_moe/config.py:100-167。通用 backend 支持检查会同时看 routing method、router logits dtype、hidden dim、activation format、batch invariance、LoRA 等条件,见 fused_moe/modular_kernel.py:536-582。因此“router 被选中”和“某 expert backend 被选中”是两步:router 先定 routing 语义,backend 再决定自己能否吃这种 routing。

vLLM 已经实现了 HPC-Ops 双 BF16 模拟 FP32 Router GEMM。本地源码没有看到双 BF16 模拟 FP32 的 router GEMM。能确认的是 DSV3/DSV4 router GEMM 支持 BF16 输入和 FP32 logits,以及 cuBLAS out_dtype=torch.float32 路径。csrc/moe/dsv3_router_gemm_entry.cu:124-127, csrc/moe/dsv4_norm_router_gemm_entry.cu:94-99, router/gate_linear.py:106-109

阅读结论

  • 确认: vLLM 的 router 选择流程集中在 router_factory.py,并在 FusedMoE.__init__ 中由模型传入的 grouped/custom/bias/hash/AITER/zero expert 参数决定。
  • 确认: 默认 top-k、bias top-k、grouped top-k 都把 topk_weights 作为 FP32 结果处理,这与 MoE router 精度敏感的需求一致。
  • 确认: GateLinear 和 NormGateLinear 是当前工作树中 router GEMM 精度优化最直接的代码入口,覆盖 DSV3/DSV4、小 token batch、BF16 输入到 FP32 logits 等路径。
  • 边界: vLLM 当前源码体现的是“BF16 输入/权重 + FP32 logits/weights”的若干实现,不是 HPC-Ops 题述“双 BF16 模拟 FP32”算法。若要移植 HPC-Ops Router GEMM,最自然的落点是 GateLinear.forward() 的 specialized/cublas tier,以及 DSV4 的 dsv4_norm_router_gemm 类似入口,而不是 expert GEMM backend oracle。
