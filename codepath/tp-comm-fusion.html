vLLM TP 通信与 RMSNorm 融合学习笔记

结论先行:普通 vLLM eager 路径里,TP all-reduce 和 RMSNorm 是两层独立机制。distributed/communication_op.py 只把 tensor 交给 TP group 的 all-reduce;CUDA communicator 再按 NCCL symmetric memory、QuickReduce、FlashInfer all-reduce、custom all-reduce、Torch SymmMem、PyNccl、PyTorch fallback 的顺序选择通信后端。残差加法与 RMSNorm 在 RMSNorm.forward() / ir.ops.fused_add_rms_norm 中完成,它可以是本地 fused add+RMSNorm kernel,但不默认包含通信。

同时,当前工作树确实存在编译期的 all-reduce+RMSNorm 融合:非 ROCm 路径可用 FlashInfer/TensorRT-LLM 风格的 flashinfer_trtllm_fused_allreduce_norm;ROCm AITER 路径可用 rocm_aiter_fused_allreduce_rmsnorm。这些路径由 FX/Inductor pattern pass 触发,受 TP size、FlashInfer/AITER 可用性、workspace、token 数和 dtype 等约束,不是 tensor_model_parallel_all_reduce() 的基础语义。

RMSNorm 实现

RMSNorm 是 CustomOp,语义是 x -> w * x / sqrt(E[x^2] + eps),见 model_executor/layers/layernorm.py:36-43。构造时它根据 IR op priority 和 variance_size_override 判断 native 路径是否需要传 weight,见 layernorm.py:47-80。CUDA forward 在 batch invariant 且无 residual 时可走 rms_norm_batch_invariant,否则回到 forward_native(),见 layernorm.py:104-116。

# vllm/model_executor/layers/layernorm.py:82-102
def forward_native(
    self,
    x: torch.Tensor,
    residual: torch.Tensor | None = None,
) -> torch.Tensor | tuple[torch.Tensor, torch.Tensor]:
    if residual is None:
        return ir.ops.rms_norm(
            x,
            self.weight.data if self.pass_weight else None,
            self.variance_epsilon,
            self.variance_size_override,
        )
    else:
        return ir.ops.fused_add_rms_norm.maybe_inplace(
            x,
            residual,
            self.weight.data if self.pass_weight_add else None,
            self.variance_epsilon,
            self.variance_size_override,
        )

IR reference 实现说明了两个核心语义:

  • rms_norm:把输入转 FP32,沿最后一维算 mean(x^2),乘 rsqrt(variance + epsilon),再乘 weight 并转回原 dtype,见 ir/ops/layernorm.py:9-21。
  • fused_add_rms_norm:先把 x + x_residual 以 FP32 相加,把相加结果写回 residual 的原 dtype 表示,再对相加后的值做 RMSNorm,见 ir/ops/layernorm.py:39-58。

CUDA-like 默认实现由平台决定。CUDA 平台在 Inductor 编译时默认优先 native,非 codegen 时默认 ["vllm_c", "native"];如果启用 Oink,会把 Oink 放到 RMSNorm priority 前面,见 platforms/cuda.py:574-593。ROCm 在满足 CUDA Graph 和 AITER env 时可把 AITER 放到 RMSNorm priority 前面,见 platforms/rocm.py:936-962。

1. 普通 eager 路径:无通信+Norm 一体化

这条路径最容易从文件边界看清楚:communication_op.py 不知道 residual 或 RMSNorm;cuda_communicator.py 的所有分支都只返回 all-reduced tensor;layernorm.py 只在拿到 tensor 后做本地 Norm。也就是说,如果没有编译期 fusion pass,一次典型的 TP 输出后处理是:

local matmul output
  -> tensor_model_parallel_all_reduce(output_parallel)
  -> materialized all-reduced tensor
  -> RMSNorm.forward(x, residual)
  -> ir.ops.fused_add_rms_norm(x, residual, weight, eps)
  -> local norm output + updated residual

2. SequenceParallelismPass:通信分解,不是单 kernel 融合

SequenceParallelismPass 明确说它识别 AllReduce 后接 RMSNorm 或 RMSNorm+Quantization,并替换为 ReduceScatter、local RMSNorm/Quantization、AllGather,见 sequence_parallelism.py:495-507。源码注释还写明这个 pass 本身不直接带来性能收益,而是为后续 GEMM + ReduceScatter 和 AllGather + GEMM 融合铺路,见 sequence_parallelism.py:508-512。

它支持无 residual 的 first-layer pattern,见 sequence_parallelism.py:150-180;也支持带 residual 的 middle-layer pattern,即 all_reduce(mm_1) 后接 fused_add_rms_norm(all_reduce, residual, weight),再改写为 reduce_scatter(mm_1)、local fused_add_rms_norm 和 all_gather,见 sequence_parallelism.py:183-257。这不是 HPC-Ops 题述的一体化 AllReduce+Norm kernel,但它把 Norm 的计算粒度从完整 sequence 改到本地 shard。

3. AllReduceFusionPass:FlashInfer fused allreduce-norm

这是当前源码里最接近 HPC-Ops 题述方案的 CUDA 路径。文件开头在 FlashInfer 可用且包含 allreduce_fusion 与 create_allreduce_fusion_workspace 时注册 custom op flashinfer_trtllm_fused_allreduce_norm,见 allreduce_rms_fusion.py:53-63 与 allreduce_rms_fusion.py:121-245。这个 custom op 调用 flashinfer_comm.allreduce_fusion(),传入 input、workspace、residual_in、residual_out、norm_out、rms_gamma、rms_eps、可选 quant 输出和 scale,见 allreduce_rms_fusion.py:194-212。

4. ROCm AITER fused allreduce-rmsnorm

ROCm 路径在 pass manager 中优先使用 RocmAiterAllReduceFusionPass:当 pass_config.fuse_allreduce_rms 为真且 AITER 启用时加入 AITER pass,否则加入 FlashInfer pass,见 compilation/passes/pass_manager.py:151-155。AITER replacement 对无 residual 和有 residual 的模式都调用 rocm_aiter_ops.get_fused_allreduce_rmsnorm_op(),见 allreduce_rms_fusion.py:913-1000。

底层 custom op _rocm_aiter_fused_allreduce_rmsnorm_impl() 根据 input dtype、hidden dim、token 数、world size 和是否 full NVLink 决定是否使用 1-stage path,然后调用 aiter_ar.fused_ar_rms(input_, residual, w=weight, eps=epsilon, ...),见 vllm/_aiter_ops.py:750-791。AITER pass 还要求 TP size 大于 1、存在 device communicator 和 custom all-reduce,并初始化 AITER allreduce,见 allreduce_rms_fusion.py:1005-1035。旧版 AITER 对 hidden dim 还有 512/1024/2048/4096 的限制检测,见 allreduce_rms_fusion.py:1042-1066。

和 HPC-Ops Fused AllReduce+Norm 对比

维度HPC-Ops 题述一体化方案vLLM 普通路径vLLM 可选融合路径
融合范围TP all-reduce + residual add + RMSNorm 一体化。all-reduce 与 RMSNorm 分离;RMSNorm 内部可融合 residual add,见 layernorm.py:82-102。FlashInfer/AITER 编译 pass 可把 all_reduce + fused_add_rms_norm 替换成 fused allreduce-norm op,见 allreduce_rms_fusion.py:347-400 与 allreduce_rms_fusion.py:959-1000。
入口 API题述上像一个专门 fused op 或 kernel。tensor_model_parallel_all_reduce(input_) 只有 input tensor 参数,见 communication_op.py:12-14。用户模型代码仍写普通 all-reduce/RMSNorm;FX/Inductor pass 做 pattern replacement。
通信后端题述未给后端细节。CUDA communicator 按 NCCL symmetric memory、QuickReduce、FlashInfer pure AR、custom AR、Torch SymmMem、PyNccl、PyTorch fallback 选择,见 cuda_communicator.py:241-298。FlashInfer fused allreduce-norm 使用 FlashInfer comm workspace;ROCm 使用 AITER allreduce。它们不走普通 communicator 的 residual-blind dispatch。
残差处理residual add 是融合 kernel 的一部分。ir.ops.fused_add_rms_norm 在本 rank 上先做 x + residual 并更新 residual,见 ir/ops/layernorm.py:39-58。FlashInfer fused op 接收 residual_in/residual_out;AITER fused op 接收 input_/residual,见 allreduce_rms_fusion.py:194-212 与 _aiter_ops.py:782-789。
RMSNorm 计算与通信同一体化调度。IR reference 用 FP32 计算方差;具体实现可落到 native、vllm_c、AITER、Oink,见 ir/ops/layernorm.py:9-58 与 kernels/vllm_c.py:22-90。FlashInfer pattern code 使用 kARResidualRMSNorm;AITER 调 fused_ar_rms。
是否支持 Norm 后量化题述只说 AllReduce+Norm,未确认量化。RMSNorm quant fusion 是另一类本地 pass,不属于普通 all-reduce。FlashInfer pass 支持 FP8 和 NVFP4 static quant pattern,见 allreduce_rms_fusion.py:419-758。
适用范围题述未给限制。所有普通 TP 路径都可走,后端自动 fallback。需要编译 pass、TP size 大于 1、FlashInfer/AITER 可用、workspace 初始化成功、token 数在上限内;AITER 还有 dtype/hidden/token/world size 等 runtime 条件。
源码可确认结论只能确认题述目标,不能确认具体实现。确认不是一体化方案。确认存在近似一体化的可选实现,但不是默认通信 API 的通用行为。

常见误读

说法判断源码依据
pynccl.py 已经做了 AllReduce+Norm fusion。不准确。PyNccl 只封装 NCCL all-reduce;NCCL symmetric memory wrapper 也只是 copy 到 symmetric memory 后 all-reduce。pynccl.py:42-48, pynccl.py:166-197

vLLM 完全没有 AllReduce+Norm fusion。也不准确。编译期确实有 FlashInfer 和 AITER 的 allreduce-rmsnorm pattern fusion。pass_manager.py:151-155, allreduce_rms_fusion.py:761-895, allreduce_rms_fusion.py:1005-1085

RMSNorm.forward(residual) 融合了 residual add。准确。有 residual 时调用 ir.ops.fused_add_rms_norm.maybe_inplace,可落到本地 fused add RMSNorm kernel。layernorm.py:95-102, kernels/vllm_c.py:54-90

阅读结论

  • 确认: distributed/communication_op.py 是很薄的 TP collective facade。它不包含 residual、RMSNorm 或 fusion 策略。
  • 确认: device_communicators/pynccl.py、symm_mem.py、quick_all_reduce.py 都是 all-reduce 通信后端或通信加速后端,不是 Norm 后端。
  • 确认: vLLM 的 RMSNorm 本身有清晰的本地计算融合:有 residual 时走 fused_add_rms_norm,语义是 residual add 后 RMSNorm。
  • 确认: 当前工作树存在通信+Norm 的可选编译期融合,包括 FlashInfer flashinfer_trtllm_fused_allreduce_norm 和 ROCm AITER rocm_aiter_fused_allreduce_rmsnorm。
  • 边界: 这些融合不是普通 all-reduce API 的默认行为。没有触发编译 pass 或后端条件不满足时,vLLM 回到 “通信先完成,再本地 residual+RMSNorm” 的分层路径。
  • 对照结论: 如果 HPC-Ops 的 Fused AllReduce+Norm 指的是“把 TP 通信、残差加法和 RMSNorm 放进一个一体化 kernel/调度单元”,vLLM 当前不是单一全局方案。它的普通路径是模块化分层;它的可选 FlashInfer/AITER pass 在满足条件时能达到相近的 fused allreduce-norm 形态;Sequence Parallelism 则是另一种把 all-reduce+norm 改写为 reduce-scatter/local norm/all-gather 的通信分解路线。
