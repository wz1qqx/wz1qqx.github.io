vLLM MoE backend 数据流与 dispatch 学习笔记

结论先行:当前工作树的 MoE 不是一个单独 kernel,而是一条由 FusedMoE、MoERunner、router、quant method、oracle selector、prepare/finalize 和 expert GEMM backend 组合出来的流水线。--moe-backend auto 也不是全局唯一优先级,而是每个量化族分别定义候选顺序,再逐个调用 expert class 的 is_supported_config() 做设备、量化、activation、parallel、routing 和 shape 检查。

Router 与 gating

router 的选择与 expert GEMM backend 是两层逻辑。router 只负责把 token 映射到专家,输出 topk_weights 和 topk_ids;后端 selector 负责决定这些 token/expert 元数据交给哪种 expert GEMM 实现。

Expert GEMM 与合并输出

MoE expert GEMM 的共同形态是两段矩阵乘:第一段 w13 计算 gate/up projection,中间 activation 或 activation+quant,第二段 w2 计算 down projection。不同 backend 的差异主要在于 token 如何 permute/dispatch、activation 是否预量化、GEMM kernel 来自 Triton/CUTLASS/FlashInfer/DeepGEMM/Marlin/Humming/AITER,以及 top-k 权重乘法和 reduction 是在 expert kernel 内还是 finalize 阶段完成。

容易误读实际情况依据
moe_backend=cutlass 是否总能选 CUTLASS?不能。FP8 显式 CUTLASS 还受 allow_vllm_cutlass 保护;unquantized selector 不接受 cutlass;W4A8 的 CUTLASS 是量化族唯一选择,不是由显式 map 控制。oracle/fp8.py:294-304, oracle/unquantized.py:131-144, oracle/w4a8.py:49-79

按文件追代码的建议顺序

  1. 先看 vllm/config/kernel.py:122-190,记住用户可配字符串只是入口名。
  2. 再看 fused_moe/layer.py:296-444,确认 router、FusedMoEConfig、quant method 和 runner 如何绑定。
  3. 看 runner/moe_runner.py:496-537 与 runner/moe_runner.py:742-799,抓住 monolithic 与普通 top-k 路径的分叉。
  4. 看 router_factory.py:63-75 与 base_router.py:250-298,确认 router 只输出 topk_weights/topk_ids。
  5. 按模型量化族进入对应 fused_moe/oracle/*.py:先读 enum、map_*_backend()、priority list,再读 convert_to_*_kernel_format() 和 make_*_moe_kernel()。
  6. 最后进 fused_moe/experts/*.py 看具体 _supports_* 和 apply();这一步才能判断某 backend 为什么在当前硬件、量化格式或并行配置下被跳过。
